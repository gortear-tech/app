# Modulo 6 - Inteligencia IA integrada y jobs
### Decisiones, memoria, estilos, prompts, captions y proveedores

## Principio

La inteligencia debe vivir en servicios internos de API y workers. El valor esta en tener decisiones trazables, ejecutadas por jobs, con datos persistidos y reglas faciles de probar.

Regla:

- La inteligencia vive como servicios internos: decision, memoria, estilo, prompts, captions, prediccion y reportes.
- La API crea comandos y jobs.
- El worker ejecuta tareas lentas o costosas.
- Supabase Queues/PGMQ puede ser la cola fisica preferida; `jobs` sigue siendo el ledger de progreso.
- Los providers solo llaman APIs externas.
- La base de datos guarda entradas, salidas, estados y eventos.
- OpenAI Responses API debe usarse con Structured Outputs para analisis, captions, planes y rankings.
- Cada salida IA estructurada debe incluir `schemaVersion`; cada prompt debe tener `promptVersion`.
- Cada llamada IA que afecte producto debe registrar un `aiRunId` con `modelProfileId`, `promptTemplateId`, `promptVersion`, `schemaVersion`, hashes de entrada/salida, `responseId`, uso, costo, latencia, `traceId` y `requestId`.
- Los providers deben registrar `model`, `responseId` si existe, `usage` sanitizado, latencia y `requestId`.
- Los providers deben propagar `traceId`/`requestId` desde API o job y adjuntar `operationKey` cuando exista.
- Para optimizar costo/latencia, mantener instrucciones y schemas estables al inicio del request y contexto variable al final para favorecer prompt caching.
- OpenAI Images API debe tener modelo primario y fallback configurables.
- OpenAI Batch/Flex solo debe usarse para jobs no interactivos donde el usuario no espera respuesta inmediata.
- Ningun cambio de modelo, prompt o schema puede entrar a produccion sin version nueva, eval minima, rollout controlado y rollback claro.

## Componentes internos recomendados

Estos componentes pueden vivir dentro de `apps/api/src/services` o en un paquete compartido neutro como `packages/domain`. Deben mantenerse pequenos, claros y ligados a contratos verificables.

| Componente | Responsabilidad |
| --- | --- |
| `decisionService` | Validar si una accion puede continuar, requiere aprobacion o queda bloqueada. |
| `autonomyService` | Calcular autonomia por accion y actualizar score con aprobaciones/rechazos. |
| `memoryService` | Construir memoria desde eventos, publicaciones y metricas. |
| `styleService` | Elegir estilo por variante y asegurar diversidad. |
| `generationPlanService` | Crear plan de generacion, riesgo, restricciones y prompt final. |
| `captionService` | Preparar contexto y validar captions generados por proveedor IA. |
| `modelRouterService` | Elegir `ModelProfile` por tarea, costo, latencia, calidad requerida y fallback. |
| `promptRegistryService` | Resolver templates versionados y separar instrucciones estables de contexto variable. |
| `qualityGateService` | Validar imagen, caption y restricciones antes de marcar una variante como revisable. |
| `evalService` | Ejecutar evaluaciones offline/canary y comparar prompts/modelos contra baseline. |
| `metricsIngestionService` | Normalizar metricas propias y de Meta sin depender de nombres de metricas fragiles. |
| `performanceSummaryService` | Crear resúmenes por post/semana/estilo con confianza estadistica. |
| `scheduleRankingService` | Ordenar variantes aprobadas por prediccion simple. |
| `weeklyReportService` | Crear resumen semanal desde eventos y metricas. |
| `jobService` | Crear, reclamar, ejecutar, cancelar y reintentar jobs. |

Reglas:

- Cada servicio recibe datos ya cargados desde la base de datos.
- Cada servicio devuelve JSON tipado y testeable.
- Ningun servicio interno conoce tokens.
- Ningun servicio interno llama Meta, OpenAI ni Supabase directamente, salvo `jobService` al coordinar providers.

## Versionado de modelos, prompts y schemas

La IA debe tratarse como software versionado, no como texto suelto en el codigo.

Contratos minimos:

- `ModelProfile`: define tarea (`vision`, `caption`, `generation_plan`, `image_generation`, `ranking`, `weekly_report`, `eval`), proveedor, modelo primario, modelo fallback, `reasoningEffort`, `textVerbosity`, schema de salida, timeout, costo maximo estimado, calidad de imagen y si permite Batch/Flex.
- `PromptTemplate`: guarda `promptTemplateId`, `promptVersion`, tarea, instrucciones estables, ejemplos permitidos, schema asociado, owner y estado (`draft`, `canary`, `active`, `retired`).
- `AiRun`: audita cada llamada con `workspaceId`, `jobId`, `operationKey`, `modelProfileId`, `promptTemplateId`, `promptVersion`, `schemaVersion`, `inputHash`, `outputHash`, `responseId`, `usage`, `costEstimate`, `latencyMs`, `status`, `safetyFlags`, `qualityStatus`, `requestId` y `traceId`.

Reglas:

- Structured Outputs es obligatorio para salidas de texto que alimentan datos de negocio: vision, captions, planes, ranking y reportes.
- El schema vive en codigo compartido y se deriva hacia JSON Schema; no se duplica a mano en prompts.
- El prompt no debe describir el schema completo si la API ya recibe Structured Outputs.
- Las instrucciones comunes, ejemplos y schema deben ir al inicio del request; datos del negocio, foto, estilo y contexto reciente van al final.
- Usar `prompt_cache_key` estable por tarea/version cuando haya prefijos largos repetidos y registrar `cached_tokens`.
- El prompt completo y respuestas crudas no deben persistirse por defecto. Persistir hashes, IDs y outputs sanitizados; permitir prompt completo solo en modo debug seguro con TTL corto.
- Si un proveedor devuelve rechazo, salida incompleta o schema invalido, el job no debe inventar datos: debe registrar fallo normalizado y decidir reintento/fallback segun `ModelProfile`.
- Cambios de `PromptTemplate` o `ModelProfile` pasan por `draft -> canary -> active`; `retired` permite rollback y reproduccion historica.

## Jobs de IA y publicacion

Las tareas lentas deben ejecutarse como jobs:

| Job | Cuando se crea | Que hace |
| --- | --- | --- |
| `analyze_photo` | Al completar upload. | Llama vision, guarda `visionAnalysis`, marca foto `validada`. |
| `generate_variant` | Al confirmar generacion. | Elige estilo, crea plan, genera imagen/caption, guarda variante. |
| `generate_batch` | Al pedir generacion de lote. | Crea jobs `generate_variant` por foto/indice y coordina progreso. |
| `schedule_posts` | Al confirmar calendario. | Ordena variantes, asigna fechas y crea scheduled posts. |
| `publish_post` | Al programar remoto, al llegar hora local o publicar ahora. | Envia a Meta de forma idempotente y guarda `facebookPostId`/`remoteStatus`. |
| `retry_post` | Al reintentar fallo. | Revalida token, evita duplicado y vuelve a publicar si procede. |
| `sync_remote_post` | Manual, periodico o tras ambiguedad. | Lee Meta para confirmar si un post existe, fue publicado, sigue programado o fallo. |
| `cancel_remote_post` | Al cancelar post que ya tiene objeto Meta. | Intenta borrar/cancelar remoto y solo marca cancelado con evidencia. |
| `reconcile_external_operation` | Al detectar timeout/caida tras iniciar proveedor. | Verifica Meta/OpenAI/Storage, actualiza entidad destino y decide si se puede reintentar. |
| `collect_metrics` | Periodico. | Consulta metricas y guarda eventos. |
| `weekly_report` | Semanal o manual. | Genera reporte de aprendizaje. |
| `batch_caption_eval` | Periodico/no urgente. | Evalua captions, estilos o historicos usando Batch/Flex para reducir costo. |

Estados de job:

- `queued`
- `running`
- `succeeded`
- `failed`
- `cancelled`
- `blocked`
- `needs_user_action`

Regla anti-duplicados:

- Cada job debe tener `dedupeKey`.
- `generate_variant` usa `variantId` o `batchId:photoId:variantIndex`.
- `publish_post` usa `scheduledPostId`.
- Si existe `facebookPostId`, no se publica de nuevo.
- Cada side effect externo debe tener `operationKey`.
- `generate_variant` usa `operationKey = openai_image:{variantId}` para imagen y `openai_caption:{variantId}` para caption si se separan.
- `publish_post` usa `operationKey = meta_publish:{scheduledPostId}`.
- `sync_remote_post` usa `operationKey = meta_sync:{scheduledPostId}:{facebookPostId}`.
- `cancel_remote_post` usa `operationKey = meta_cancel:{scheduledPostId}:{facebookPostId}`.
- `retry_post` reutiliza el mismo `scheduledPostId` y revisa `ExternalOperation` antes de llamar Meta.
- `reconcile_external_operation` usa `dedupeKey = operationKey` para que solo exista una reconciliacion activa por side effect.
- Si el proveedor soporta idempotency keys, enviar `operationKey` o un derivado estable; si no, usar reconciliacion.

## Reclamo concurrente de jobs

Cuando haya mas de un worker, el reclamo debe usar una de estas dos rutas:

Ruta preferida con Supabase Queues/PGMQ:

- worker lee mensajes con visibility timeout;
- mensaje trae `jobId`;
- worker carga job desde DB;
- worker crea `JobAttempt` antes de llamadas externas;
- al terminar persiste entidades, `JobAttempt`, `ExternalOperation` y despues archiva/elimina mensaje;
- si worker cae, el mensaje reaparece para otro worker.

Fallback con tabla `jobs`:

- seleccionar jobs `queued` vencidos por `runAfter`;
- bloquear fila con `FOR UPDATE SKIP LOCKED`;
- marcar `running`, `lockedAt` y `lockedBy` en la misma transaccion;
- asignar `leaseExpiresAt`;
- ejecutar fuera de la transaccion larga;
- al terminar, marcar `succeeded`, `failed`, `blocked` o `needs_user_action`.

Esto permite paralelismo sin que dos workers ejecuten el mismo job.

## Reintentos y ambiguedad

Los jobs se dividen por riesgo:

| Riesgo | Jobs | Politica |
| --- | --- | --- |
| Bajo costo/reversible | `collect_metrics`, `weekly_report`, evaluaciones batch | Reintento automatico con backoff. |
| Costo medio | `analyze_photo`, `generate_variant`, captions | Reintento automatico si no empezo proveedor; si empezo y quedo ambiguo, reconciliar asset/resultado. |
| Alto riesgo externo | `publish_post`, editar/cancelar remoto | No reintentar a ciegas; reconciliar Meta y mantener `estado_incierto` si no hay certeza. |
| Reconciliacion | `reconcile_external_operation` | Sin side effects nuevos salvo lectura/verificacion; actualiza estado solo con evidencia. |

Reglas:

- Backoff exponencial con jitter por tipo de job.
- Timeouts obligatorios por proveedor.
- Errores 4xx funcionales no se reintentan salvo reconexion/cambio de permisos.
- Errores 429/5xx se reintentan con backoff y limite.
- Si el usuario cancela lote/post, los jobs pendientes pasan a `cancelled`; jobs `running` deben revisar estado antes de escribir.
- Cualquier job que detecte entidad cerrada debe salir como `cancelled` o `blocked`, no revivirla.

## Outbox para eventos

Todo job o comando que cambie entidades debe escribir eventos de salida en outbox dentro de la misma transaccion.

Uso:

- registrar auditoria confiable;
- alimentar memoria/autonomia;
- disparar notificaciones futuras;
- sincronizar con una cola externa si aparece.

Regla:

- Primero se confirma DB; despues un worker procesa el outbox. Nunca al reves.

## Decision service

Entrada:

- usuario/owner;
- negocio;
- batch;
- accion solicitada;
- costo estimado;
- autonomia;
- riesgo;
- estado de token;
- permisos de pagina.

Salida:

```ts
type DecisionResult = {
  outcome: "allowed" | "requires_approval" | "blocked";
  reasonCode: string;
  userMessage?: string;
  riskLevel: "low" | "medium" | "high";
  autonomyUsed: boolean;
};
```

Bloquea si:

- batch esta `cancelado`, `fallido` o `abandonado`;
- falta token de pagina;
- el costo no esta confirmado;
- el job duplicaria una publicacion;
- el proveedor requerido no esta configurado.

## Autonomy service

Cada accion conserva:

- score;
- approvals;
- rejections;
- threshold;
- paused;
- consecutiveApprovals;
- consecutiveRejections.

Acciones:

- `STYLE_ASSIGNMENT`
- `VARIANT_COUNT`
- `SCHEDULING`
- `CAPTION_GENERATION`
- `FACEBOOK_PUBLISH`

Reglas:

- aprobaciones repetidas suben autonomia;
- rechazos la bajan;
- riesgo alto impide autonomia;
- poca historia impide autonomia total;
- publicar en Facebook requiere confirmacion hasta que exista opt-in explicito, historial suficiente, token sano, presupuesto configurado y reglas de horario aprobadas.
- `FACEBOOK_PUBLISH` nunca puede volverse autonoma solo por aprobaciones implicitas; requiere accion dedicada del usuario en Configuracion.
- cualquier deteccion de acceso Meta expirado, post incierto, costo fuera de presupuesto, persona visible o promocion/precio visible pausa autonomia de publicacion.

## Memory service

Construye memoria desde base de datos, no desde un snapshot en memoria.

Fuentes:

- eventos de dominio;
- variantes aprobadas/rechazadas;
- captions editados;
- posts publicados/fallidos;
- metricas recolectadas;
- snapshots de metricas normalizados;
- resúmenes de performance con confianza;
- configuracion vigente del negocio.

Calcula:

- estilos preferidos;
- tipos de contenido preferidos;
- horarios con mejor desempeno;
- patrones de captions editados;
- tasa de aprobacion por estilo;
- tasa de publicacion exitosa;
- senales de engagement por post cuando Meta las entregue;
- comparaciones por ventana temporal equivalente;
- confianza por tamano de muestra.

Regla:

- Nunca vender causalidad fuerte con pocos datos. Usar confianza `exploratoria`, `media` o `alta`.
- No comparar posts con ventanas de vida distintas como si fueran equivalentes. Usar ventanas normalizadas: 24h, 72h, 7d y lifetime.
- Si Meta retira/renombra una metrica, el aprendizaje debe degradar con gracia y usar metricas propias/eventos internos.
- Un post no debe influir en autonomia ni ranking hasta tener ventana minima o sample suficiente.

## Style service

El estilo se asigna por variante, no por foto.

Entrada:

- negocio;
- SEO;
- foto con `visionAnalysis`;
- estilos vigentes;
- memoria;
- variantes hermanas;
- indice de foto;
- indice de variante.

Puntuacion:

- match con industria;
- match con SEO y contexto local;
- match con sujeto, mood y composicion;
- prioridad inicial a estilos `realista_seguro` y `editorial_comercial`;
- estilos `experimental` solo con opt-in, bajo riesgo y buena compatibilidad;
- penalizacion fuerte si el estilo ya se uso en la misma foto;
- penalizacion moderada si ya se uso en el lote;
- penalizacion si intensidad fuerte toca logos, personas, precios o texto visible;
- desempate determinista.

Salida:

- `AssignedStyle` guardado completo dentro de `Variant`.

## Quality gate de variantes

Antes de pasar una variante a `generada`, el worker debe ejecutar una compuerta de calidad. Esta compuerta no decide gusto creativo; decide si el resultado es publicable y honesto.

Validaciones de imagen:

- existe `MediaAsset` generado privado;
- formato y MIME permitido;
- proporcion 1:1 o recorte/resize controlado;
- resolucion minima definida por producto;
- no contiene metadata sensible innecesaria;
- respeta restricciones de `GenerationPlan`;
- no altera logos, precios, texto visible ni identidad de personas cuando estaban protegidos;
- si se necesita publicar en Meta, existe o puede crearse `publishableAssetId`.

Validaciones de caption:

- espanol natural;
- una o dos frases;
- maximo dos hashtags;
- usa SEO sin keyword stuffing;
- no repite inicios recientes;
- no promete descuentos, precios, disponibilidad o resultados que no vienen del negocio;
- no contradice lo que aparece en la imagen.

Salida:

```ts
type AiQualityCheck = {
  schemaVersion: "ai_quality_check.v1";
  status: "pass" | "warn" | "block";
  score: number;
  warnings: string[];
  blockingReasons: string[];
  requiresHumanReview: boolean;
};
```

Reglas:

- `block` deja la variante `fallida` o `bloqueada_por_calidad` si se agrega ese estado.
- `warn` permite revision humana, pero no autopublicacion.
- Solo `pass` puede entrar al flujo normal de aprobacion sin friccion extra.
- El resultado se guarda como `AiQualityCheck` y se referencia desde `Variant`.

- captions recientes a evitar.

Salida:

```ts
type CaptionResult = {
  schemaVersion: "caption.v1";
  caption: string;
  seoTermsUsed: string[];
  warnings: string[];

```

Reglas:

- espanol natural;

## Providers IA

`VisionAnalysisProvider`

- `analyze(imageUrl)`
- Devuelve `VisionAnalysisResult` estricto.

`ImageGenerationProvider`

- `generateImage({ prompt, sourceImageUrl })`
- Devuelve imagen en data URL o binario temporal para subir a Storage.

`CaptionGenerationProvider`

- `generateCaption(context)`
- Devuelve JSON estricto.

`EvaluationProvider`

- `runEval({ evalId, promptTemplateId, modelProfileId, datasetId })`
- Devuelve metricas agregadas, criterios fallidos y URL/referencia de reporte si existe.

Reglas:

- Providers no conocen batch, negocio completo ni tokens de Meta.
- Providers no persisten.
- Providers reportan errores normalizados.
- API/worker convierten errores en estados de job y `AppErrorResponse`.
- Providers deben exponer modo sincronico para interaccion directa y modo asincrono/batch solo para tareas que acepten demora.

## Evaluaciones y rollout de IA

La calidad de IA se valida con evaluaciones repetibles, no solo mirando ejemplos buenos.

Dataset minimo:

- fotos de comida, belleza, servicios, retail y locales;
- fotos con personas visibles;
- fotos con logos, precios, menus, texto y promociones;
- imagenes oscuras, borrosas o con mala composicion;
- negocios con SEO muy amplio y muy especifico;
- captions historicos aprobados, editados y rechazados.

Metricas:

- tasa de schema valido;
- tasa de rechazo/refusal;
- tasa de variantes bloqueadas por calidad;
- tasa de captions editados por usuario;
- duplicacion de inicios de caption;
- costo por variante aprobada;
- latencia p50/p95 por tarea;
- tasa de aprobacion por estilo/modelo;
- incidentes de claims comerciales no soportados.

Reglas de rollout:

- Cada prompt/modelo nuevo corre eval offline contra golden set antes de canary.
- Canary inicial recomendado: 10% a 20% de workspaces internos/testers.
- Activar solo si mejora o empata calidad y no rompe costo/latencia.
- Si sube la tasa de bloqueo, edicion manual o fallos de schema, volver al perfil anterior.
- Batch/Flex se usa para evaluaciones, reportes y regeneraciones no urgentes; no para el flujo donde el usuario espera aprobar variantes en vivo.

## Programacion inteligente

El ranking de variantes aprobadas debe ser simple y auditable:

- prioridad a estilos con mayor aprobacion historica;
- variedad de estilos en dias consecutivos;
- horarios preferidos del negocio;
- evitar concentrar variantes de la misma foto seguidas;
- penalizar contenido con riesgo o bajo confidence.

Salida:

- lista ordenada de `variantId`;
- razon corta por variante;
- fecha/hora sugerida.

Reglas de prediccion:

- La prediccion solo ordena; no bloquea contenido aprobado.
- Usar features estables: `styleId`, tipo de foto, horario local, dia de semana, longitud de caption, hashtags, industria y performance historica normalizada.
- No usar metricas de Meta que esten deprecated como unica senal.
- Con menos de 20 posts publicados del negocio, usar heuristicas conservadoras y confianza `exploratoria`.
- Con 20-99 posts, permitir ajustes suaves por estilo/horario.
- Con 100+ posts y metricas sanas, permitir recomendaciones con confianza `media` o `alta`.

## Reporte semanal

El reporte se genera desde eventos y metricas.

Secciones:

- que funciono;
- que no funciono;
- estilos con mejor aceptacion;
- captions mas editados;
- horarios recomendados;
- salud de publicacion Meta;
- cobertura de calendario;
- costo IA usado vs presupuesto;
- proximas recomendaciones.

Regla:

- Reporte no debe inventar metricas. Si falta data, decir que la muestra aun es pequena.
- Reporte debe separar "acciones dentro de FBmaniaco" de "metricas reportadas por Meta".
- No mostrar ranking de mejor horario/estilo si la muestra no alcanza el minimo.
- Toda recomendacion debe incluir `confidence` y `reasonCode`.

## Metricas y aprendizaje

El aprendizaje profesional se basa en dos capas:

1. Datos propios confiables: aprobaciones, rechazos, ediciones, publicaciones, fallas, horarios, costos, estados remotos y cobertura semanal.
2. Datos de Meta variables: insights/post metrics cuando existan, versionados por nombre de metrica y fecha de recoleccion.

Contratos:

- `MetricDefinition`: catalogo interno que mapea una metrica canonica (`views`, `engagements`, `reactions`, `comments`, `shares`, `clicks`, `publish_success`) a proveedor, nombre remoto, version Graph y estado (`active`, `deprecated`, `unavailable`).
- `PostMetricSnapshot`: valor observado para un post en una ventana (`24h`, `72h`, `7d`, `lifetime`), con `providerMetricName`, `canonicalMetric`, `value`, `collectedAt` y `collectionStatus`.
- `PerformanceSummary`: resumen por negocio/semana/estilo/horario con sample size, metricas normalizadas y confianza.

Reglas:

- `collect_metrics` no debe fallar todo el job por una metrica de Meta invalida; marca esa metrica `unavailable/deprecated` y guarda evidencia.
- Las metricas crudas se guardan como snapshots inmutables; los summaries se pueden recalcular.
- Si una metrica cambia de significado o desaparece, se crea nueva `MetricDefinition`; no se sobreescribe historia.
- La app nunca muestra "este estilo funciona mejor" sin `sampleSize`, ventana y confianza.
- Los reportes no deben sumar organic/paid si el producto solo gestiona organic Page posts.
- Para MVP, el aprendizaje puede ser util aun sin Insights: aprobar/rechazar, editar caption, publicar/fallar y cobertura semanal ya son senales suficientes.

## Interfaz con otros modulos

- API crea comandos y jobs.
- Worker ejecuta jobs y actualiza estados.
- Lotes consume progreso y resultados.
- Calendario consume ranking y jobs de publicacion.
- Configuracion alimenta SEO, estilos y autonomia.
- Datos guarda jobs, eventos, variantes, captions y metricas.
- Seguridad define que datos pueden llegar a providers y UI.

## Reglas de reconstruccion

- No crear paquetes de inteligencia con frontera artificial si los servicios internos bastan.
- Crear servicios internos pequenos, testeables y sin marca propia.
- Crear tabla `jobs` desde el inicio.
- Ejecutar vision, generacion, scheduling, publicacion y metricas por jobs.
- Mantener providers aislados.
- Guardar todas las decisiones importantes en eventos.
- Persistir en base de datos antes de responder exito.
- Usar snapshot/cache solo como respaldo, no como fuente primaria.
