AUTONOMY MANAGER — DISEÑO DE MÓDULO
Motor Perron · Capa de Núcleo
Versión 1.0 | Abril 2026
================================================================================

Este documento describe el diseño completo del módulo AutonomyManager dentro de
Motor Perron. Es un documento de diseño funcional: no contiene código, describe
comportamiento, reglas y decisiones de diseño.


────────────────────────────────────────────────────────────────────────────────
1. QUÉ ES Y PARA QUÉ SIRVE
────────────────────────────────────────────────────────────────────────────────

AutonomyManager es el módulo que decide cuándo el sistema puede actuar sin
pedirle permiso al usuario y cuándo debe presentar una propuesta para que el
humano la apruebe o rechace.

Su objetivo es que con el tiempo el sistema se vuelva progresivamente más
autónomo, reduciendo la fricción para el usuario sin que este pierda el control
real de lo que se publica. La autonomía no se configura manualmente: el sistema
la gana a través de resultados consistentes. El usuario nunca tiene que definir
umbrales ni entender cómo funciona internamente.

Lo que sí puede hacer el usuario es ver qué acciones ya maneja el sistema solo
y cuáles todavía requieren su aprobación, y puede bajar la confianza de una
acción específica si no está conforme con cómo el sistema la está manejando.


────────────────────────────────────────────────────────────────────────────────
2. LAS CINCO ACCIONES QUE ADMINISTRA
────────────────────────────────────────────────────────────────────────────────

AutonomyManager administra la confianza de forma independiente para cinco tipos
de acción. Cada una tiene su propio score, su propio historial y su propio
umbral de autonomía interno:

  STYLE_ASSIGNMENT
    Asignar el estilo visual a una foto automáticamente sin mostrarlo al usuario
    para que lo confirme.

  VARIANT_COUNT
    Decidir cuántas variantes generar por foto sin preguntarle al usuario.

  SCHEDULING
    Elegir la fecha y hora exacta de cada publicación dentro del período
    seleccionado por el usuario, sin mostrar el calendario para confirmación.

  CAPTION_GENERATION
    Generar el caption de cada publicación sin presentarlo al usuario para
    que lo revise antes de aprobar.

  FACEBOOK_PUBLISH
    Publicar directamente en Facebook cuando llega la hora programada, sin
    ninguna confirmación adicional del usuario.

Estas cinco acciones tienen distinto nivel de riesgo inherente. STYLE_ASSIGNMENT
es la de menor riesgo porque el usuario siempre ve el resultado antes de publicar.
FACEBOOK_PUBLISH es la de mayor riesgo porque una vez publicado no hay vuelta
atrás fácil. El sistema toma esto en cuenta en sus umbrales internos.


────────────────────────────────────────────────────────────────────────────────
3. CÓMO SE GANA LA AUTONOMÍA
────────────────────────────────────────────────────────────────────────────────

El sistema gana autonomía en una acción cuando acumula suficiente evidencia de
que sus decisiones en esa acción producen buenos resultados de forma consistente.

SCORE DE CONFIANZA

Cada acción tiene un score de confianza que va de 0 a 100. El sistema comienza
con score 0 en todas las acciones. Los umbrales internos de autonomía son:

  STYLE_ASSIGNMENT   → umbral: 60 puntos
  VARIANT_COUNT      → umbral: 65 puntos
  SCHEDULING         → umbral: 70 puntos
  CAPTION_GENERATION → umbral: 75 puntos
  FACEBOOK_PUBLISH   → umbral: 85 puntos

Los umbrales más altos para las acciones de mayor riesgo reflejan que el sistema
necesita más evidencia antes de actuar solo en decisiones que tienen consecuencias
más difíciles de revertir.

REQUISITO MÍNIMO DE APROBACIONES

Además del score, cada acción requiere un mínimo de aprobaciones acumuladas antes
de poder actuar autónomamente, sin importar cuál sea el score:

  STYLE_ASSIGNMENT   → mínimo 8 aprobaciones
  VARIANT_COUNT      → mínimo 10 aprobaciones
  SCHEDULING         → mínimo 12 aprobaciones
  CAPTION_GENERATION → mínimo 15 aprobaciones
  FACEBOOK_PUBLISH   → mínimo 20 aprobaciones

Esto evita que el sistema gane autonomía con pocos datos aunque todos sean
positivos. Un negocio que aprueba todo en sus primeros 5 lotes no le ha dado
al sistema suficiente variedad de situaciones para confiar en él.

CÓMO SUBE EL SCORE

El score sube cuando el usuario aprueba en el swipe de autonomía una propuesta
del sistema para esa acción:
  Aprobación simple → +4 puntos
  Aprobación con resultado medido positivo (post que rindió bien) → +6 puntos
  Aprobación consecutiva sin interrupciones → +1 punto adicional por racha

CÓMO BAJA EL SCORE

El score baja cuando el usuario rechaza una propuesta del sistema o cuando
el usuario solicita bajar la confianza de una acción específica:
  Rechazo en el swipe → -8 puntos (el doble que una aprobación)
  Rechazo consecutivo → -12 puntos (señal de patrón problemático)
  Reset manual de esa acción por el usuario → score vuelve a 0

El rechazo pesa más que la aprobación porque una mala decisión autónoma tiene
más impacto negativo que el valor de una buena decisión. Este principio es
estándar en sistemas de confianza de plataformas como los algoritmos de
moderación de contenido de redes sociales.


────────────────────────────────────────────────────────────────────────────────
4. EL SWIPE DE AUTONOMÍA
────────────────────────────────────────────────────────────────────────────────

Cuando el sistema quiere ejecutar una acción pero su score no ha alcanzado el
umbral de autonomía, en lugar de bloquear el flujo presenta una propuesta al
usuario en formato de swipe.

El swipe de autonomía funciona igual que el swipe de aprobación de variantes:
una tarjeta por propuesta, deslizar a la derecha para aprobar, a la izquierda
para rechazar. Cada aprobación o rechazo actualiza el score de confianza de
esa acción específica.

CONTENIDO DE CADA TARJETA

La tarjeta muestra la propuesta del sistema en lenguaje simple y directo, sin
tecnicismos. Ejemplos:

  Para STYLE_ASSIGNMENT:
    "Para esta foto de tus platillos elegí el estilo Producto apetitoso.
     ¿Te parece bien?"

  Para SCHEDULING:
    "Programé esta publicación para el jueves a las 7pm. Ese horario ha
     funcionado muy bien para ti últimamente."

  Para CAPTION_GENERATION:
    "Generé este caption para la foto del lote. ¿Lo apruebas tal como está?"

El usuario no necesita entender por qué el sistema propone lo que propone.
La tarjeta es simple y la decisión es binaria: sí o no.

CUÁNDO DESAPARECE EL SWIPE

Una vez que el score de una acción supera su umbral y se cumple el mínimo de
aprobaciones, esa acción deja de aparecer en el swipe. El sistema la ejecuta
directamente sin interrumpir al usuario. El usuario lo nota porque esa parte
del flujo simplemente ya no aparece.


────────────────────────────────────────────────────────────────────────────────
5. PROTECCIONES PERMANENTES
────────────────────────────────────────────────────────────────────────────────

Independientemente del score de confianza acumulado, ciertas situaciones siempre
requieren aprobación humana. Estas protecciones no pueden desactivarse:

ELEMENTOS SENSIBLES
  Cualquier foto con logos, precios visibles, personas o marcas detectadas por
  el análisis de visión siempre requiere revisión humana antes de publicar.
  El score de FACEBOOK_PUBLISH puede ser 100 y aun así estas fotos pasan por
  aprobación. No hay excepción.

PRIMER LOTE DE UN NEGOCIO NUEVO
  El primer lote de cualquier negocio siempre pasa por aprobación completa en
  todas las acciones, sin importar el historial de otros negocios del mismo
  usuario. Cada negocio empieza desde cero.

CAÍDA DE RENDIMIENTO DETECTADA
  Si DeepMemory detecta una ruptura de patrón (rendimiento cayendo más del 30%
  en los últimos 30 días), AutonomyManager pausa automáticamente la autonomía
  en SCHEDULING y FACEBOOK_PUBLISH hasta que el sistema acumule 5 nuevas
  aprobaciones en esas acciones. Es una pausa de seguridad automática, no un
  reset permanente.

RECONEXIÓN DE FACEBOOK
  Después de que el token de Facebook expira y se reconecta, FACEBOOK_PUBLISH
  requiere al menos 3 aprobaciones manuales antes de volver a actuar de forma
  autónoma, independientemente del score previo.


────────────────────────────────────────────────────────────────────────────────
6. VISIBILIDAD PARA EL USUARIO
────────────────────────────────────────────────────────────────────────────────

El usuario puede ver en cualquier momento qué acciones ya maneja el sistema de
forma autónoma y cuáles todavía requieren su participación. Esta información
está disponible en la configuración del negocio en una sección simple llamada
algo como "Lo que el sistema ya hace solo".

CÓMO SE PRESENTA

La pantalla muestra las cinco acciones con su estado actual en lenguaje simple:

  ✓ Asignar estilo a tus fotos → El sistema ya lo hace solo
  ✓ Decidir cuántas variantes generar → El sistema ya lo hace solo
  ○ Elegir horarios de publicación → Todavía te pide confirmación
  ○ Generar captions → Todavía te pide confirmación
  ○ Publicar en Facebook → Todavía te pide confirmación

No se muestran números, scores ni porcentajes. Solo el estado: autónomo o
requiere confirmación. El usuario no necesita entender cómo funciona el sistema
para tomar decisiones sobre él.

BOTÓN DE RESET POR ACCIÓN

Junto a cada acción que ya es autónoma aparece un botón discreto que permite
bajar su confianza si el usuario no está conforme con cómo el sistema la maneja.
Al tocar el botón, el score de esa acción vuelve a cero y las próximas ejecuciones
de esa acción vuelven al swipe de aprobación.

El reset es por acción individual, no global. Si el usuario está conforme con
que el sistema asigne estilos solo pero no con que elija horarios, puede resetear
solo SCHEDULING sin afectar las demás.


────────────────────────────────────────────────────────────────────────────────
7. CÓMO SE RELACIONA CON LOS DEMÁS MÓDULOS
────────────────────────────────────────────────────────────────────────────────

decisionEngine
  Es el principal consumidor de AutonomyManager. Antes de devolver su veredicto
  final consulta a AutonomyManager para cada tipo de acción involucrada en la
  tarea. Si el score no supera el umbral, el outcome cambia de
  "puede_continuar_autonomo" a "puede_continuar_swipe".

DeepMemory
  AutonomyManager recibe de DeepMemory la señal de ruptura de patrón cuando
  aplica, para activar la pausa automática de seguridad en SCHEDULING y
  FACEBOOK_PUBLISH.

API
  La API registra cada aprobación y rechazo del swipe de autonomía como
  BusinessLearningEvent de tipo accion_aprobada o accion_rechazada. Esos
  eventos alimentan el score de confianza en la siguiente llamada a
  AutonomyManager.

  La API también expone el estado de autonomía por acción al frontend para
  que la pantalla de configuración del negocio pueda mostrarlo.


────────────────────────────────────────────────────────────────────────────────
8. REGLAS IRROMPIBLES
────────────────────────────────────────────────────────────────────────────────

  1. Los elementos sensibles siempre requieren aprobación humana. Esta regla
     no puede ser superada por ningún score de confianza, sin excepción.

  2. Cada negocio empieza con score 0 en todas las acciones. El historial
     de otros negocios del mismo usuario no transfiere confianza.

  3. El score nunca puede superar 100 ni bajar de 0.

  4. El rechazo siempre pesa el doble que la aprobación en el cálculo del score.

  5. El mínimo de aprobaciones es un requisito independiente del score. Ambas
     condiciones deben cumplirse para que el sistema actúe autónomamente.

  6. El reset es siempre por acción individual. No existe un reset global que
     borre la confianza de todas las acciones a la vez.

  7. AutonomyManager nunca le explica al usuario cómo funciona su sistema de
     scoring. Solo muestra el estado resultante: autónomo o requiere confirmación.

  8. La pausa automática por caída de rendimiento afecta solo a SCHEDULING y
     FACEBOOK_PUBLISH, nunca a las acciones de menor riesgo.
