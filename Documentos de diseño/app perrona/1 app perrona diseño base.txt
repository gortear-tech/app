Este documento describe la plataforma App Perrona en su concepto, arquitectura y
funcionamiento interno con suficiente detalle para que pueda ser reconstruida desde
cero en cualquier tecnología. No contiene secretos ni credenciales de servicios
externos. El motor de inteligencia (Motor Perron) tiene su propio documento de
diseño; aquí se describe únicamente su interfaz y sus puntos de integración con
la plataforma.


────────────────────────────────────────────────────────────────────────────────
1. VISIÓN Y PROPÓSITO
────────────────────────────────────────────────────────────────────────────────

App Perrona es una plataforma de marketing en redes sociales orientada a pequeños
y medianos negocios. Su propuesta central es reducir el trabajo manual necesario
para publicar contenido de calidad en Facebook de forma consistente.

El usuario fotografía sus productos o local, los sube a la plataforma, y el sistema
se encarga de analizarlos individualmente con IA de visión, asignarles un estilo
visual personalizado, editarlos, generar textos descriptivos (captions), proponer
un calendario de publicación y finalmente publicar automáticamente en los horarios
acordados.

Con el tiempo, la plataforma aprende qué tipos de contenido funcionan mejor para
cada negocio y aumenta su nivel de autonomía progresivamente, reduciendo la
intervención humana necesaria. El objetivo a largo plazo es que el único paso que
requiera al humano sea tomar las fotos y subirlas. Todo lo demás, el motor lo gana
progresivamente.

La plataforma también educa al dueño del negocio: cada semana recibe un reporte
en lenguaje simple que explica qué funcionó, por qué, y qué hacer la semana
siguiente. El negocio no solo se automatiza, también comprende su propia estrategia.

El producto actual es un MVP orientado a un único usuario operador que gestiona
uno o varios negocios conectados a sus páginas de Facebook.


────────────────────────────────────────────────────────────────────────────────
2. ESTRUCTURA GENERAL (MONOREPO)
────────────────────────────────────────────────────────────────────────────────

El proyecto está organizado como un monorepo gestionado con pnpm workspaces y
Turborepo como orquestador de build y tareas. Esto permite compartir código entre
aplicaciones con total seguridad de tipos.

Estructura de carpetas raíz:

  apps/
    api/          → Backend REST (servidor Fastify)
    mobile/       → App móvil multiplataforma (Expo / React Native)
    worker/       → Proceso de tareas asíncronas (colas BullMQ)

  packages/
    shared/       → Tipos, contratos HTTP, estados, constantes, errores
    providers/    → Adaptadores para servicios externos (contratos + mocks)
    motor-perron/ → Motor de inteligencia interna (paquete desacoplado)

  supabase/       → Migraciones de base de datos secundaria y funciones serverless
  docs/           → Documentación técnica interna

Lenguaje base: TypeScript 5.x en toda la plataforma.
Módulos: ESNext con resolución NodeNext.
Target de compilación: ES2022.
Path aliases configurados:
  @app-perrona/shared        → packages/shared
  @app-perrona/providers     → packages/providers
  @app-perrona/motor-perron  → packages/motor-perron

Tests: Jest con soporte ESM en todas las apps y paquetes.

Estructura interna de motor-perron (packages/motor-perron/src/):
  core/           → decisionEngine, orchestrator, autonomyManager, types
  intelligence/   → deepMemory, causalAnalyzer, performancePredictor, benchmarkEngine
  strategy/       → contentStrategyEngine, weeklyReporter, recommendationEngine,
                    recommendations
  content/        → styleAssigner, promptBuilder
  rules/          → rules
  index.ts        → barrel de exportaciones públicas

3. APLICACIONES
────────────────────────────────────────────────────────────────────────────────

3.1 API (apps/api)
──────────────────

PROPÓSITO
  Backend central de la plataforma. Expone una API REST/JSON que consume la app
  móvil. Gestiona autenticación, negocios, lotes de fotos, análisis de visión,
  generación de variantes con IA, programación y publicación en redes sociales.

TECNOLOGÍA
  - Framework HTTP: Fastify v5
  - ORM: Prisma v6 (PostgreSQL)
  - Autenticación: JWT (access tokens) + refresh tokens con HMAC
  - Tipos de contrato: Paquete @app-perrona/shared

ARQUITECTURA INTERNA
  El API sigue un patrón de módulos por dominio. Cada módulo tiene:
    - *.service.ts   → lógica de negocio pura (sin dependencia de HTTP)
    - *.routes.ts    → registro de rutas Fastify, validación de entrada, mapeo a servicio
    - *.test.ts      → tests de integración usando repositorios en memoria

  Módulos existentes:
    auth/          → Registro, login, refresh, logout, sesiones
    businesses/    → CRUD de negocios, dashboard inteligente, reporte semanal
    batches/       → Ciclo de vida de lotes de fotos
    variants/      → Generación, aprobación y rechazo de variantes
    scheduling/    → Calendario de publicaciones
    vision/        → Análisis visual de fotos (delegado a proveedor de IA de visión)
    metrics/       → Recolección y scoring de rendimiento de posts
    me/            → Datos del usuario autenticado
    http/          → Middleware de autenticación

  Patrón repositorio:
    La API nunca accede directamente a Prisma desde los servicios. Usa interfaces
    de repositorio definidas en repositories/repository.types.ts. Hay dos
    implementaciones intercambiables:
      - prismaRepositories.ts   → Producción (PostgreSQL)
      - inMemoryRepositories.ts → Tests (sin base de datos)

    Esto permite correr todos los tests sin instancia de base de datos.

PUNTO DE ENTRADA
  src/main.ts inicializa Fastify y registra todos los módulos.
  src/app.ts configura middlewares, plugins y rutas.
  src/config.ts valida y carga todas las variables de entorno al arrancar.


3.2 MOBILE (apps/mobile)
─────────────────────────

PROPÓSITO
  Aplicación cliente multiplataforma (iOS, Android, Web) con la que el usuario
  interactúa con la plataforma. El diseño prioriza la inmersión y la simplicidad:
  cada pantalla tiene una sola tarea clara y las interacciones son gestuales y
  directas.

TECNOLOGÍA
  - Framework: Expo v52 / React Native v0.76
  - Web: React Native Web v0.19
  - Almacenamiento local seguro: expo-secure-store
  - Selector de imágenes: expo-image-picker
  - Bundler: Metro (configuración estándar Expo)
  - Comunicación con backend: HTTP REST contra la API

ARQUITECTURA INTERNA
  src/AppShell.tsx               → Shell principal; decide qué pantalla mostrar
  src/authFlow.ts                → Lógica de decisión de pantalla inicial
                                   (¿existe usuario? ¿hay sesión activa? ¿token vivo?)
  src/sessionStorage.ts          → Persistencia local de accessToken y refreshToken
                                   usando almacenamiento seguro del dispositivo
  src/businessSettingsStorage.ts → Persistencia local de negocio seleccionado
  src/facebookReconnect.ts       → Lógica del flujo de reconexión de red social

  La app consulta al API al arrancar para saber si existe un usuario registrado
  (endpoint bootstrap). Según el resultado, navega a registro o a login.

  No existe router declarativo complejo; la navegación se maneja con estado en
  AppShell que decide el árbol de componentes a renderizar.

PANTALLAS Y FLUJOS PRINCIPALES

  1. Carga inicial → decisión automática (registro o login)
  2. Registro de usuario único
  3. Login
  4. Pantalla principal del negocio
     - Lista de lotes con su estado actual
     - Vista compacta del calendario con puntos de color
     - Tarjeta destacada del lote activo con el paso exacto en que quedó
     - Botón prominente para subir lote nuevo
     - Botón de configuración
     - Estado cero: si no hay lotes, el espacio de la lista muestra una
       invitación a subir el primer lote con el botón grande centrado
  5. Subida de fotos (ver sección 9.3 para flujo detallado)
  6. Thumbnails con estilos asignados y cambio rápido de estilo
  7. Detalle opcional de foto individual
  8. Selector de cantidad de variantes
  9. Pantalla de generación con mensajes en vivo
  10. Swipe de aprobación / rechazo de variantes
  11. Resumen y selección de período de publicación
  12. Calendario de publicaciones
  13. Reconexión de página de red social

COMPORTAMIENTOS TRANSVERSALES DE LA APP
  - Banner fijo en la parte baja mientras el análisis de visión corre en background.
    Muestra progreso tipo "Analizando 4 de 6 fotos..." con barra de avance. Al
    terminar cambia a "Lote listo · Ver fotos" con un toque para ir directo.
  - El lote activo a medias siempre es accesible desde la pantalla principal del
    negocio con el paso exacto donde quedó.
  - Las notificaciones push informan al usuario cuando: el análisis terminó, las
    variantes están listas, una publicación falló en Facebook, y el reporte
    semanal está disponible.
  - Los colores del calendario indican el estado de cada publicación de un vistazo:
    naranja → programada, verde → publicada, rojo → fallida, gris → pausada.

VARIABLES DE ENTORNO ESPERADAS (prefijo EXPO_PUBLIC_)
  - URL base de la API
  - Configuración de la base de datos secundaria (para lectura de páginas públicas)
  - Identificador de la app de red social (para flujo OAuth)
  - URI de redirección del flujo OAuth


3.3 WORKER (apps/worker)
─────────────────────────

PROPÓSITO
  Proceso Node.js independiente que ejecuta tareas asíncronas en segundo plano.
  Se encarga de publicar posts cuando llega su hora programada, limpiar lotes
  abandonados, recolectar métricas de Facebook, y disparar los ciclos semanales
  de estrategia y reporte. No expone HTTP; consume colas de mensajes.

TECNOLOGÍA
  - Sistema de colas: BullMQ v5 (sobre Redis)
  - Runtime: Node.js con TypeScript (ejecutado con tsx)
  - Compartido: @app-perrona/shared para tipos de jobs

MODOS DE EJECUCIÓN
  El worker soporta tres modos configurables en runtime:

    idle                    → Se inicia pero no procesa nada (útil para pruebas)
    publish-due-once        → Procesa un lote de posts pendientes y termina
    cleanup-abandoned-once  → Marca lotes abandonados y termina

  Parámetros configurables:
    publishBatchSize       → Cantidad de posts a procesar por ejecución (default: 20)
    abandonedBatchSize     → Cantidad de lotes a evaluar por ejecución (default: 50)

TAREAS

  A) Publicación programada (publishing.ts)
     Consulta posts en estado "programada" cuya fecha/hora ya llegó.
     Para cada uno, invoca al proveedor de publicación externo.
     Actualiza el estado del post según el resultado.
     Si falla, el post queda en estado "fallida" o "pausada_por_token" y se
     genera una alerta visible en el calendario con opción de reintentar.
     También se envía notificación push al usuario.
     Dato del job: { scheduledPostId, negocioId, trigger, requestedAt }
     Trigger puede ser "schedule" (automático) o "manual" (usuario).

  B) Limpieza de lotes abandonados (abandonedBatches.ts)
     Busca lotes en estado "pending_upload" o "pendiente_confirmacion" con más
     de 7 días sin actividad.
     Los marca como "abandonado".
     Previene acumulación de datos huérfanos en la base de datos.

  C) Recolección de métricas
     Consulta a Facebook las métricas de posts publicados recientemente.
     Calcula el score de rendimiento interno.
     Guarda el PostPerformance correspondiente.
     Registra un BusinessLearningEvent de tipo post_measured.

  D) Brief semanal (cada lunes)
     Llama a contentStrategyEngine de Motor Perron con el historial del negocio.
     Recibe el brief de contenido de la semana.
     Envía notificación push al usuario con el resumen.

  E) Reporte semanal (cada domingo)
     Llama a causalAnalyzer, benchmarkEngine y weeklyReporter de Motor Perron.
     Recibe el reporte completo de la semana.
     Lo persiste y lo hace visible en el dashboard del negocio.
     Envía notificación push al usuario.

COLAS DEFINIDAS (queues.ts)
  Una o más colas con nombre constante compartido entre quien encola (API) y quien
  consume (Worker). Los nombres son strings exportados desde queues.ts.

4. PAQUETES COMPARTIDOS

4.1 SHARED (packages/shared)
──────────────────────────────

PROPÓSITO
  Fuente única de verdad para todos los tipos, contratos HTTP, estados, constantes
  y errores de la plataforma. Es el único canal de comunicación de tipos entre el
  API y la app móvil. No tiene dependencias de runtime; solo tipos TypeScript y

MÓDULOS EXPORTADOS

  api.ts
    ApiRouteMap: mapeo exhaustivo de todas las rutas HTTP con tipos de request y
    response. Permite que el cliente y el servidor compartan el mismo contrato de

  apiClient.ts
    Helper de cliente HTTP que construye requests tipados a partir del ApiRouteMap.

  auth.ts
    Tipos: PublicUser, AuthSession, BootstrapStatusResponse
    Requests/Responses: RegisterInitialRequest/Response, LoginRequest/Response,

  business.ts

      - alerts: alertas activas del negocio (incluyendo fallos de Facebook)
      - activeBatch: resumen del lote en proceso con paso actual
      - upcomingPosts: próximos posts programados
      - recentPosts: posts publicados recientemente
      - performance: métricas de rendimiento reciente
      - recommendations: recomendaciones de Motor Perron
      - weeklyReport: reporte semanal si está disponible
      - weeklyBrief: brief de contenido si es lunes
    Requests: CreateBusinessRequest, UpdateBusinessRequest

  batches.ts
    Tipos: BatchSummary, BatchDetail, GetBatchResponse
    Photo requests: PreparePhotoUploadRequest, CompletePhotoUploadRequest,
    CorrectPhotoClassificationRequest

  variants.ts
    Tipos: GeneratedVariant, VariantSummary
    Requests: UpdateVariantCaptionRequest
    Responses: GenerateBatchVariantsResponse, ApproveVariantResponse,
    RejectVariantResponse

  scheduling.ts
    Tipos: ScheduledPostSummary, SuggestCalendarRequest, UpdateScheduledPostRequest
    Responses: ConfirmCalendarResponse, CancelScheduledPostResponse,
    PublishScheduledPostResponse

  vision.ts
    Tipos del análisis visual de una foto:
      VisionAnalysisResult    → Resultado completo del análisis (inmutable tras guardarse)
      VisionAnalysisSubject   → Qué hay en la foto (tipo, descripción, si hay personas)
      VisionAnalysisComposition → Encuadre, ángulo, tipo de fondo
      VisionAnalysisPalette   → Tonos dominantes, temperatura de color
      VisionAnalysisSensitiveElements → Precios, logos, marcas, personas identificables
      VisionAnalysisTechnicalQuality  → Nitidez, exposición, ruido
      VisionAnalysisMood      → Temperatura emocional de la imagen
      AssignedStyle           → Estilo asignado con parámetros de edición resultantes
                                (editable por el usuario antes de generar)
    Enums:
      VisionSubjectType: producto | persona | comida | lugar | animal | objeto
      VisionFraming: primer_plano | plano_medio | plano_general | detalle | cenital
      VisionAngle: frontal | picado | contrapicado | lateral | cenital
      VisionBackgroundType: limpio | natural | urbano | interior | exterior | abstracto
      VisionLighting: natural | artificial | mixta | baja_luz | contraluz
      VisionMoodTemperature: calida | neutra | fria | vibrante | oscura

      id, name, description, promptTemplate, recommendedIndustries,
      recommendedPhotoTypes, intensity, aiDisclosureRequired, restrictions
    INITIAL_VISUAL_STYLES: catálogo de 6 estilos predefinidos:
      1. Corrección de iluminación    → Ajuste de luz y exposición
      2. Mejora realista              → Realce fiel al producto
      3. Producto apetitoso           → Para fotografía de comida
      4. Fondo limpio                 → Eliminar distracciones del fondo
      5. Estilo cálido                → Tonos cálidos y acogedores
      6. Estilo premium               → Look de alta gama

  states.ts
    Todos los enums de estado de la plataforma (ver sección 6).

  metrics.ts
    Tipos: BusinessPerformanceSummary, DashboardRecommendation,
    MetricsCollectionWindow, PostPerformanceMetrics

  errors.ts
    Clase AppError con campos: code, statusCode, message, userMessage, details
    Usada en toda la plataforma para errores esperados con contexto estructurado.

  costs.ts
    Tipos para desglose de costos operativos de generación de variantes.

  facebook.ts
    Tipos para la integración con la red social (páginas, tokens, publicaciones).


4.2 PROVIDERS (packages/providers)
────────────────────────────────────

PROPÓSITO
  Define los contratos (interfaces) para todos los servicios externos que usa la
  plataforma, y provee implementaciones mock para desarrollo y testing. Las
  implementaciones reales de producción también residen aquí como adaptadores.

  La idea es que el API y el Worker nunca dependan directamente de un SDK externo
  en su lógica de negocio. Siempre usan una interfaz de este paquete.

CONTRATOS DEFINIDOS

  VisionAnalysisProvider
    analyze(imageUrl: string): Promise<VisionAnalysisResult>
    Analiza una imagen y devuelve el resultado estructurado del análisis visual.
    El proveedor real usa GPT-4o Vision. El proveedor se selecciona mediante la
    variable de entorno VISION_PROVIDER (openai | mock).

  MediaStorage
    generateSignedUploadUrl(key: string): Promise<{ url: string; key: string }>
    getPresignedDownloadUrl(key: string): Promise<string>
    Abstrae el almacenamiento de archivos; el API genera URLs firmadas que el
    cliente usa para subir/leer archivos directamente sin pasar por el servidor.

  FacebookPublishingProvider
    publishPost(data: PublishPostData): Promise<{ postId: string }>
    getPageAccessToken(pageId: string): Promise<string>
    getPageMetrics(pageId: string, postId: string): Promise<MetricsData>
    Abstrae la publicación en la red social y la recolección de métricas.

IMPLEMENTACIONES MOCK (mocks.ts)
  MockVisionAnalysisProvider     → Devuelve un análisis simulado predecible
  MockMediaStorage               → Simula URLs firmadas sin llamada real
  MockFacebookPublishingProvider → Simula publicación sin llamada real a red social

  Los mocks permiten correr el 100% de la lógica de negocio en tests y en
  desarrollo sin credenciales ni dependencias externas.

IMPLEMENTACIONES DE PRODUCCIÓN
  openaiVision.ts          → Implementación real del análisis visual con IA
  supabaseFacebook.ts      → Lectura de tokens de páginas y publicación real


4.3 MOTOR PERRON (packages/motor-perron)
──────────────────────────────────────────

PROPÓSITO
  Motor de inteligencia interna de la plataforma. Toma decisiones sobre qué hacer
  con el contenido, qué estilo aplicar, si puede actuar de forma autónoma o necesita
  aprobación humana, y genera recomendaciones, reportes y estrategia de contenido
  basados en el historial de cada negocio.

  Principio fundamental: Motor Perron nunca conecta a bases de datos ni hace llamadas
  HTTP. Recibe datos como parámetros y devuelve decisiones como objetos TypeScript.
  Es una biblioteca pura de lógica de negocio.

  Ver documento "motor-perron-arquitectura.txt" para el diseño completo interno.

INTERFAZ CON LA PLATAFORMA (puntos de integración)

  A) Al analizar una foto recién subida (síncrono, bloquea el flujo de subida):
     API llama a styleAssigner con el resultado de VisionAnalysis y el perfil del
     negocio. Recibe el estilo asignado y los parámetros de edición.
     API llama a promptBuilder con el estilo y la foto. Recibe el prompt final.
     API llama a autonomyManager para saber si puede proceder autónomamente o
     necesita cola de swipe para esa acción específica.
     Si el análisis de visión falla o vence el timeout, la foto no se guarda y
     el usuario debe reintentar la subida.

  B) Al generar variantes de un lote:
     API llama a decisionEngine con el lote, las fotos y el contexto del negocio.
     DecisionEngine devuelve un MotorDecision que indica:
       - Si puede continuar y en qué modalidad (autónomo o cola de swipe)
       - El modo de IA a usar (ligera | avanzada)
       - Si requiere aprobación humana obligatoria (elementos sensibles)
       - El nivel de riesgo (bajo | medio | alto)
       - Recomendaciones de acción
     Si la decisión es positiva, API usa el GenerationPlan de cada foto para
     construir el request al proveedor de IA.

  C) Al cargar el dashboard de un negocio:
     API llama a deepMemory con el historial del negocio para construir los
     3 modelos de memoria (rendimiento estadístico, huella del negocio, mapa causal).
     API llama a recommendationEngine para obtener recomendaciones accionables.

  D) Ciclo semanal de estrategia (cada lunes):
     Worker llama a contentStrategyEngine para generar el brief de contenido.
     El brief se envía como notificación push al usuario.

  E) Ciclo semanal de reporte (cada domingo):
     Worker llama a causalAnalyzer para procesar los eventos de la semana.
     Worker llama a benchmarkEngine para obtener datos del grupo par.
     Worker llama a weeklyReporter para generar el reporte completo.
     El reporte se persiste y se envía como notificación push.

  F) Al programar posts:
     API puede llamar a performancePredictor para estimar el rendimiento esperado
     de un post antes de confirmarlo en el calendario. Este score se usa
     internamente para optimizar el horario; el usuario no lo ve.
     API puede llamar a benchmarkEngine para comparar el negocio contra su grupo par.

TIPOS CLAVE DE LA INTERFAZ

  MotorDecision {
    negocioId: string
    taskType: AiTaskType
    mode: "sin_ia" | "ia_ligera" | "ia_avanzada"
    outcome: "puede_continuar_autonomo" | "puede_continuar_swipe" |
             "requiere_aprobacion" | "bloqueado"
    confidenceLevel: ConfidenceLevel
    riskLevel: "riesgo_bajo" | "riesgo_medio" | "riesgo_alto"
    reason: string
    requiresHumanApproval: boolean
    recommendedActions: string[]
  }

  GenerationPlan {
    puedeGenerar: boolean
    motivo: string
    sujetoPrincipal: string
    preservar: string[]    → Qué NO modificar en la imagen
    permitido: string[]    → Qué SÍ puede cambiar la IA
    prohibido: string[]    → Qué está absolutamente prohibido
    riesgo: string[]       → Elementos sensibles detectados
    nivelRiesgo: RiskLevel
    divulgacionIa: "no_requerida" | "recomendada" | "obligatoria"
    promptFinal: string
    promptVersion: string
    planVersion: string
  }

  ActionType (enum del AutonomyManager)
    STYLE_ASSIGNMENT    → asignar estilo visual a una foto
    VARIANT_COUNT       → decidir cuántas variantes generar por foto
    SCHEDULING          → elegir el horario de publicación
    CAPTION_GENERATION  → generar el texto de la publicación
    FACEBOOK_PUBLISH    → publicar directamente en Facebook

  BusinessLearningEvent {
    negocioId: string
    type: BusinessLearningEventType
    occurredAt: string        → ISO 8601
    styleId?: string
    styleName?: string
    photoType?: string
    captionPattern?: string
    scheduledFor?: string
    score?: number
    costUsd?: number
    provider?: string
    model?: string
  }

  Los BusinessLearningEvents son registrados por la API cada vez que ocurre algo
  relevante. Motor Perron los recibe como array en sus llamadas y aprende de ellos
  sin persistencia propia.


────────────────────────────────────────────────────────────────────────────────
5. MODELO DE DATOS (PostgreSQL / Prisma)
────────────────────────────────────────────────────────────────────────────────

La base de datos principal es PostgreSQL gestionada con Prisma ORM. A continuación
se describen las entidades del modelo con sus campos más relevantes.

USER
  id              UUID, clave primaria
  email           string único
  passwordHash    string (scrypt + pepper)
  status          UserStatus enum
  createdAt       timestamp
  updatedAt       timestamp
  lastLoginAt     timestamp opcional
  → Tiene: sessions, businesses, auditLogs

SESSION
  id              UUID, clave primaria
  userId          FK → User
  refreshTokenHash string (HMAC del refresh token real)
  deviceInfo      string opcional (user-agent u otro identificador)
  expiresAt       timestamp
  revokedAt       timestamp opcional (null = activa)

BUSINESS (negocio)
  id              UUID, clave primaria
  userId          FK → User (pertenencia)
  nombre          string
  giroComercial   string (tipo de industria o actividad)
  facebookPageId  string único (id de la página de red social)
  tokenStatus     FacebookTokenStatus enum
  timezone        string (zona horaria del negocio)
  metadata        JSON opcional (datos adicionales configurables)
  autonomySettings JSON (umbrales por ActionType definidos por el usuario)
  → Tiene: batches, scheduledPosts, auditLogs

BATCH (lote de fotos)
  id                  UUID, clave primaria
  negocioId           FK → Business
  status              BatchStatus enum
  photosCount         int (fotos totales)
  variantsCount       int (variantes generadas)
  estimatedCostUsd    float opcional
  confirmedCostUsd    float opcional
  lastActivityAt      timestamp
  → Tiene: originalPhotos, generatedVariants, scheduledPosts

ORIGINAL_PHOTO
  id                  UUID, clave primaria
  batchId             FK → Batch
  storageProvider     string (nombre del proveedor de almacenamiento)
  fileUrlInternal     string (ruta interna en el almacenamiento)
  visionAnalysis      JSON opcional (VisionAnalysisResult completo, inmutable)

────────────────────────────────────────────────────────────────────────────────
8. AUTENTICACIÓN Y SEGURIDAD
────────────────────────────────────────────────────────────────────────────────

MODELO DE SESIÓN
  La plataforma usa un esquema de doble token:

  Access Token (JWT):
    - Vida corta (minutos / pocas horas)
    - Firmado con clave secreta (SESSION_SECRET)
    - Contiene: userId, sessionId, expiresAt
    - Enviado en cada request como Bearer Token
    - No se almacena en base de datos; se verifica contra la sesión activa

  Refresh Token:
    - Vida larga (días / semanas)
    - Opaco y aleatorio (no decodificable)
    - Almacenado en el cliente en secure storage del dispositivo
    - Se guarda en base de datos SOLO su hash HMAC (nunca el token en texto plano)
    - Al usar el refresh token, se rota: se genera uno nuevo y el anterior se invalida
    - Se puede revocar por sessionId (logout)

CONTRASEÑAS
  Algoritmo: scrypt
  Pepper: variable de entorno adicional al salt (doble protección)
  El hash nunca se expone en ninguna respuesta de la API

UNICIDAD DEL USUARIO
  La plataforma está diseñada para un único usuario principal (operador).
  El endpoint /auth/register-initial solo funciona si no existe ningún usuario.
  Intentar registrar un segundo usuario retorna error.

AUDITORÍA
  Toda acción relevante genera un registro en AuditLog con:
    - Quién (userId)
    - Sobre qué negocio (negocioId)
    - Qué acción
    - Qué entidad afectada
    - Estado anterior y nuevo (como JSON)

  Acciones auditadas: login, logout, creación de negocio, subida de foto,
  cambio de estilo asignado, aprobación/rechazo de variante, publicación,
  reintento de post fallido, cambios de configuración de autonomía.

PERTENENCIA DE RECURSOS
  Cada request que accede a un negocio, lote, foto o variante verifica que:
    1. El usuario esté autenticado (access token válido)
    2. La sesión esté activa (no revocada)
    3. El negocio pertenezca al usuario autenticado
    4. El lote/foto/variante pertenezca al negocio solicitado
  Cualquier violación devuelve 403 Forbidden.

TOKENS DE RED SOCIAL
  Los tokens de acceso a la red social nunca se envían al cliente móvil.
  El cliente solo ve identificadores públicos (page_id, page_name).
  Los tokens se leen en el servidor desde el almacenamiento seguro en el momento
  de publicar. El cliente no tiene forma de extraerlos.

9. FLUJOS PRINCIPALES DE NEGOCIO
────────────────────────────────────────────────────────────────────────────────

9.1 PRIMER ARRANQUE
  1. Cliente consulta GET /auth/bootstrap-status
  2. Si hasUsers = false → mostrar pantalla de registro
  3. POST /auth/register-initial con email y contraseña
  4. Respuesta incluye AuthSession con accessToken y refreshToken
  5. Cliente almacena refreshToken en secure storage
  6. Redirige a flujo de creación de primer negocio

9.2 LOGIN Y RENOVACIÓN DE SESIÓN
  1. POST /auth/login → devuelve AuthSession
  2. Cliente usa accessToken en cada request (Bearer)
  3. Si API responde 401 (token expirado):
     a. POST /auth/refresh con refreshToken
     b. Recibe nuevo accessToken y refreshToken
     c. Guarda nuevo refreshToken, descarta el anterior
     d. Reintenta el request original
  4. Si el refresh también falla → redirige a login

9.3 CICLO COMPLETO DE UN LOTE

  Paso 1 — Creación
    POST /businesses/:id/batches → estado: pending_upload
    El usuario llega desde el botón grande de la pantalla principal del negocio.

  Paso 2 — Subida de fotos y análisis automático
    a. POST .../photos/upload-intent → API devuelve URL firmada
    b. Cliente sube archivo directamente al almacenamiento usando la URL firmada
    c. POST .../photos/complete-upload → API registra foto en estado "analizando"
    d. API llama síncronamente al VisionAnalysisProvider (GPT-4o Vision)
       dentro del timeout configurado (VISION_ANALYSIS_TIMEOUT_MS)
    e. Si el análisis falla o vence el timeout:
       → Se elimina el registro de la foto
       → Se devuelve error tipado al cliente
       → El usuario debe reintentar la subida
    f. Si el análisis tiene éxito:
       → Se persiste visionAnalysis (inmutable)
       → Motor Perron styleAssigner → asigna assignedStyle
       → Motor Perron promptBuilder → construye editingPrompt
       → Se persiste assignedStyle, editingPrompt y visionAnalyzedAt
       → Foto pasa a estado: validada
    g. Mientras el análisis de cada foto corre, la app muestra un banner fijo
       en la parte inferior con el progreso. El usuario puede navegar libremente.
    h. Repetir pasos a-g para cada foto del lote.

  Paso 3 — Revisión de thumbnails y estilos
    a. Al terminar el análisis, el banner cambia a "Lote listo · Ver fotos"
    b. El usuario ve un grid de thumbnails con el nombre del estilo asignado
    c. Toque largo sobre un thumbnail abre un menú pequeño justo encima con el
       catálogo de estilos. Cada opción muestra nombre y una línea descriptiva.
       Al cambiar el estilo, el editingPrompt se regenera automáticamente.
    d. Toque simple en un thumbnail abre el detalle completo de esa foto:
       análisis de IA con etiquetas, parámetros del estilo en barras de
       porcentaje, y el prompt de edición completo.
    e. Si el prompt está regenerando (por cambio de estilo), el detalle
       muestra "Actualizando prompt..." en ese espacio.

  Paso 4 — Confirmación de costo y cantidad de variantes
    a. GET .../style-suggestions → Motor Perron sugiere estilos para el lote
    b. POST .../estimate-cost → Motor Perron calcula costo estimado
    c. El usuario elige cuántas variantes generar por foto (selector numérico,
       total calculado en tiempo real)
    d. POST .../confirm-cost → estado lote: confirmado

  Paso 5 — Generación
    a. POST .../generate → API llama a decisionEngine de Motor Perron
    b. Si outcome = "bloqueado" → retorna error con motivo, pantalla de error
       con botón para reintentar el lote completo
    c. Si outcome = "puede_continuar_autonomo" o "puede_continuar_swipe":
       - Para cada foto: API usa el editingPrompt almacenado para llamar al
         proveedor de IA
       - Proveedor genera imagen editada
       - Proveedor genera caption
       - API guarda GeneratedVariant con estado: generada
    d. Estado lote: completado (o generado_parcial si hubo fallos)
    e. El usuario puede quedarse en la pantalla de generación viendo mensajes
       fugaces en tiempo real, o salir y recibir una notificación push cuando
       el lote esté listo.

  Paso 6 — Swipe de aprobación / rechazo
    a. Cliente carga GET .../variants
    b. La imagen ocupa casi toda la pantalla. El caption aparece arriba con
       degradado oscuro suave. Tocar el caption o deslizar hacia abajo lo
       despliega completo; deslizar hacia arriba lo contrae.
    c. Deslizar derecha o tocar palomita verde → variante: aprobada
       Deslizar izquierda o tocar equis roja → variante: rechazada
    d. Un botón pequeño de deshacer en la esquina superior revierte el último
       swipe. No hay un límite de deshaceres.
    e. Un indicador de posición muy pequeño arriba muestra la foto y variante
       actual: "F2 · V1 de 3". El swipe fluye sin interrupciones entre fotos.
    f. Las variantes se presentan agrupadas por foto: primero todas las
       variantes de la foto 1, luego todas las de la foto 2, etc.
    g. Al aprobar o rechazar la última variante del lote, aparece una
       animación breve de confeti antes de pasar al resumen.
    h. El usuario puede editar el caption de cualquier variante antes de
       aprobar. La edición se registra como BusinessLearningEvent (caption_edited).

  Paso 7 — Período de publicación
    a. El resumen muestra el número grande de variantes aceptadas
    b. El usuario elige el período en que se distribuirán: 7, 14 o 30 días
       (con una línea indicando la frecuencia resultante)
    c. POST .../calendar/confirm → crea ScheduledPosts en estado: programada
       La distribución y optimización de horarios ocurre internamente usando
       el performancePredictor de Motor Perron; el usuario no la ve.

  Paso 8 — Calendario
    a. El usuario ve el mes con puntos de color: naranja (programada), verde
       (publicada), rojo (fallida), gris (pausada)
    b. Tocar una publicación abre su detalle donde puede editar fecha y hora
       o cancelarla
    c. Si una publicación falla en Facebook, aparece alerta roja sobre ese
       día con botón de reintentar. También llega notificación push.

  Paso 9 — Publicación automática
    a. Worker detecta posts cuya scheduledFor llegó
    b. Worker usa FacebookPublishingProvider para publicar
    c. Éxito: ScheduledPost pasa a estado: publicada, guarda facebookPostId
    d. Error de token: pasa a estado: pausada_por_token, genera alerta en dashboard
    e. Error de permiso: ídem, con tipo de alerta diferente

  Paso 10 — Métricas
    a. Worker recolecta métricas del post publicado (alcance, interacciones)
    b. Calcula score interno
    c. Guarda PostPerformance
    d. API registra BusinessLearningEvent → Motor Perron aprende en la próxima consulta

9.4 DASHBOARD DEL NEGOCIO
  GET /businesses/:id/dashboard llama internamente a Motor Perron con:
    - Historial de BusinessLearningEvents del negocio
    - Posts recientes y próximos
    - Métricas de rendimiento

  Motor Perron construye los 3 modelos de memoria y devuelve:
    - Alertas activas (token expirado, lote abandonado, publicación fallida, etc.)
    - Resumen de rendimiento reciente
    - Recomendaciones accionables (qué tipo de foto subir, qué horario, etc.)
    - Brief de contenido semanal (si es lunes, generado por contentStrategyEngine)
    - Reporte semanal (si está disponible, generado el domingo anterior)

9.5 CICLO DE APRENDIZAJE SEMANAL
  Lunes: Worker llama a contentStrategyEngine → brief semanal → notificación push
    "Esta semana necesitas: 2 fotos de producto, 1 momento de equipo."
  Domingo: Worker llama a causalAnalyzer + benchmarkEngine + weeklyReporter
    → reporte semanal → se persiste → notificación push → visible en dashboard

9.6 SWIPE DE AUTONOMÍA
  Cuando Motor Perron determina que una acción no supera el umbral de confianza
  configurado por el usuario para ese tipo de acción (ActionType), la API agrega
  la propuesta a una cola de swipe. La app la presenta como una tarjeta con la
  propuesta del motor. El usuario desliza para aprobar o rechazar.
  La aprobación actualiza el score de confianza del autonomyManager hacia arriba.
  El rechazo lo baja con mayor peso.
  Elementos sensibles (precios, logos, personas, promociones) siempre van a
  aprobación humana sin importar el score de confianza.

10. COMUNICACIÓN ENTRE COMPONENTES
────────────────────────────────────────────────────────────────────────────────

App Mobile  ←→  API
  Protocolo: HTTP/1.1 REST, JSON
  Autenticación: Bearer Token (JWT)
  Contrato: ApiRouteMap de @app-perrona/shared (tipado estático compartido)
  El cliente usa el helper apiClient.ts para construir requests tipados

API  ←→  Worker
  Protocolo: Colas de mensajes (BullMQ sobre Redis)
  La API encola jobs; el Worker los consume
  Contrato: tipos ScheduledPublishingJobData de @app-perrona/shared
  Redis es el broker de mensajes (no hay comunicación HTTP entre API y Worker)

API  ←→  Motor Perron
  Protocolo: Llamadas de función directas (mismo proceso)
  Motor Perron es un paquete importado, no un servicio separado
  No hay red, no hay serialización: la API llama funciones TypeScript puras

API  ←→  Providers
  Protocolo: Llamadas de función mediante interfaces tipadas
  El API importa contratos de @app-perrona/providers
  En tests/desarrollo: mocks
  En producción: implementaciones reales del proveedor configurado

Worker  ←→  Motor Perron
  Igual que API ←→ Motor Perron. El Worker también importa el paquete
  directamente para los ciclos semanales de estrategia y reporte.

Worker  ←→  Providers
  Ídem que API ←→ Providers para la publicación

12. SISTEMA DE ESTILOS VISUALES
────────────────────────────────────────────────────────────────────────────────

Los estilos visuales son la unidad de intención de edición de la plataforma.
Un estilo describe qué tipo de transformación visual se le aplica a una foto.

Cada estilo contiene:
  - id y nombre descriptivo
  - description: qué hace en lenguaje humano (visible en el menú de cambio rápido)
  - promptTemplate: instrucción base para la IA generativa (con variables a rellenar)
  - recommendedIndustries: lista de giros comerciales donde aplica bien
  - recommendedPhotoTypes: tipos de foto donde el estilo funciona mejor
  - intensity: "ligera" | "media" | "fuerte"
  - aiDisclosureRequired: si la imagen editada requiere indicar que fue modificada por IA
  - restrictions: qué no está permitido hacer con este estilo

Los 6 estilos iniciales cubren los casos más comunes de negocios locales:
  1. Corrección de iluminación  → intensity: ligera, aiDisclosureRequired: false
  2. Mejora realista            → intensity: ligera, aiDisclosureRequired: false
  3. Producto apetitoso         → intensity: media, para comida
  4. Fondo limpio               → intensity: media, elimina distracciones,
                                   aiDisclosureRequired: true (modifica el fondo)
  5. Estilo cálido              → intensity: media, tonos acogedores
  6. Estilo premium             → intensity: fuerte, aiDisclosureRequired: true

El catálogo vive en @app-perrona/shared/styles.ts y puede extenderse.
La tabla Style en base de datos refleja este catálogo y permite adiciones futuras.

El menú de cambio rápido de estilo en los thumbnails muestra, para cada opción,
el nombre y una línea corta describiendo el look. Ejemplo:
  "Producto apetitoso · colores ricos, foco en textura y detalle"
  "Fondo limpio · elimina distracciones, protagonismo al producto"


────────────────────────────────────────────────────────────────────────────────
13. ESTRUCTURA PARA RECIBIR A MOTOR PERRON
────────────────────────────────────────────────────────────────────────────────

La plataforma está diseñada desde el inicio para que Motor Perron sea un componente
opcional e intercambiable. Si Motor Perron no estuviera disponible o se quisiera
reemplazar, los siguientes contratos son los únicos puntos de integración:

FUNCIONES QUE LA PLATAFORMA LLAMA A MOTOR PERRON

  styleAssigner(foto: VisionAnalysisResult, negocio: BusinessProfile, estilos: VisualStyle[])
    → AssignedStyle
    Elige el estilo más apropiado para una foto dado el contexto del negocio.
    Con suficiente historial, personaliza basándose en la huella del negocio.

  promptBuilder(style: AssignedStyle, foto: VisionAnalysisResult)
    → string (prompt final para el proveedor de IA)
    Construye la instrucción de edición a partir del estilo y el análisis.
    Se llama de nuevo si el usuario cambia el estilo asignado.

  autonomyManager.canProceedAutonomously(negocioId, taskType, historial)
    → { autonomous: boolean, requiresSwipe: boolean }
    Decide si puede actuar sin aprobación humana para este tipo de tarea.
    Siempre retorna requiresSwipe: true si hay elementos sensibles.

  autonomyManager.recordApproval(negocioId, taskType)
    → void
    Registra una aprobación en el swipe de autonomía para actualizar el score.

  autonomyManager.recordRejection(negocioId, taskType)
    → void
    Registra un rechazo en el swipe de autonomía para actualizar el score.

  decisionEngine.decide(lote, fotos, negocio, historial)
    → MotorDecision
    Decisión central sobre si puede generar, en qué modo, con qué riesgo.

  deepMemory.build(historial: BusinessLearningEvent[])
    → { performanceModel, businessFootprint, causalMap }
    Construye los 3 modelos de memoria del negocio.

  recommendationEngine.generate(memoryModels, negocio)
    → DashboardRecommendation[]
    Genera recomendaciones accionables para el dashboard en tiempo real.

  contentStrategyEngine.generateWeeklyBrief(memoryModels, negocio)
    → ContentBrief
    Brief semanal de qué contenido crear. Se llama cada lunes.

  weeklyReporter.generate(memoryModels, negocio, benchmarks, causalMap)
    → WeeklyReport
    Reporte dominical del desempeño de la semana en lenguaje simple.

  performancePredictor.predict(variant, scheduledFor, negocio, memoryModels)
    → { estimatedScore: number, confidence: ConfidenceLevel }
    Estimación de rendimiento antes de publicar. Uso interno, no visible al usuario.

  benchmarkEngine.compare(negocio, memoryModels)
    → BenchmarkResult
    Comparación contra grupo par de negocios similares (misma industria,
    tamaño y zona geográfica). Datos anónimos y agregados.

  causalAnalyzer.analyze(historial, rules)
    → CausalMap
    Detecta qué variables realmente mueven el rendimiento usando estadística
    y reglas predefinidas. Sin IA generativa: resultados deterministas.

    - variante_generada

DATOS QUE MOTOR PERRON NUNCA NECESITA
  - Credenciales o secrets de ningún tipo
  - Acceso a base de datos o HTTP
  - Información de sesión o autenticación
  - Configuración de infraestructura

PRINCIPIO DE PREPARACIÓN
  Cualquier reconstrucción de la plataforma solo necesita:
  1. Implementar las funciones de la lista anterior devolviendo los tipos correctos
  2. Registrar BusinessLearningEvents en los momentos indicados
  3. Llamar al motor en los momentos del ciclo: foto, generación, dashboard,
     lunes (brief) y domingo (reporte)
  El resto de la plataforma (API, Mobile, Worker, modelo de datos) es independiente
  del motor y puede funcionar con implementaciones stub mientras el motor se desarrolla.

14. TESTING
────────────────────────────────────────────────────────────────────────────────

Cada app y paquete tiene su propio jest.config.mjs y puede correr tests de forma
independiente con "pnpm test" desde su carpeta o desde la raíz del monorepo.

Estrategia de testing:
  - API: Tests de integración por módulo usando repositorios en memoria
    (inMemoryRepositories). No requieren base de datos real. Cubren rutas completas,
    incluyendo el flujo de análisis de visión con MockVisionAnalysisProvider.
  - Mobile: Tests de lógica de flujo (authFlow.test.ts). No cubren UI.
  - Worker: Tests de lógica de procesamiento con mocks de colas.
  - Shared: Tests de validación de enums y constantes.
  - Providers: Tests de mocks.
  - Motor Perron: Tests de reglas (rules.test.ts).

Los mocks de @app-perrona/providers permiten que todos los servicios que usan
proveedores externos sean completamente testeables sin credenciales.

15. REGLAS DE DISEÑO
────────────────────────────────────────────────────────────────────────────────

Estas reglas fueron establecidas en la arquitectura del proyecto y deben
preservarse en cualquier reimplementación:

  1. El cliente móvil nunca recibe secretos ni tokens de servicios externos.

  2. Motor Perron es una biblioteca pura: sin estado persistido, sin red, sin DB.
     Todo lo que necesita se le pasa como parámetro; todo lo que produce se devuelve
     como valor de retorno.

  3. Los proveedores externos (análisis visual, publicación, almacenamiento) siempre
     se acceden a través de contratos de interfaz. Nunca se llama un SDK directamente
     desde la lógica de negocio.

  4. Los tests de la API no requieren base de datos real. Los repositorios en memoria
     son ciudadanos de primera clase, no hacks de testing.

  5. Los tipos compartidos entre API y Mobile son la única fuente de verdad del
     contrato HTTP. Si el tipo cambia en shared, el compilador detecta los puntos
     de ruptura en ambos lados.

  6. Cada decisión de Motor Perron se persiste en base de datos en su forma completa
     y estructurada (GenerationPlan como JSON). Esto permite auditoría, debugging y
     aprendizaje retrospectivo.

  7. Un negocio solo puede tener un lote activo a la vez. Crear un nuevo lote
     requiere que el anterior esté en estado terminal.

  8. La auditabilidad es obligatoria: toda acción que modifique estado de negocio,
     foto, variante o publicación genera un AuditLog con el estado anterior y el nuevo.

  9. El análisis de visión es síncrono y bloquea el flujo de subida. Si falla,
     la foto no existe. No se guarda ninguna foto sin su análisis completo,
     su estilo asignado y su prompt de edición.

  10. El campo visionAnalysis de una foto es inmutable una vez escrito. El usuario
      puede cambiar el assignedStyle y editingPrompt, pero nunca el análisis raw.

  11. Los elementos sensibles (precios, logos, personas, promociones) siempre
      requieren aprobación humana, sin importar el nivel de confianza del motor.
      Esta regla no puede desactivarse desde configuración.

  12. El usuario define los umbrales de autonomía por tipo de acción desde la
      configuración. El motor puede actuar solo cuando supera el umbral Y ha
      acumulado al menos 10 aprobaciones previas en esa categoría.

  13. Los datos de benchmarks entre negocios son siempre anónimos y agregados.
      Ningún negocio puede ver datos individuales de otro, solo promedios del
      grupo par.

  14. El score de rendimiento predicho por performancePredictor es de uso interno
      exclusivo. Nunca se expone al usuario en ninguna interfaz.
